# 前后端边界与云端部署方案

## 1. 本周约束变化

老师已明确模型部分可以放在云端，因此当前不再强行要求模型在小程序本地运行。新的工程目标是：

- 小程序端保持轻量，只负责拍摄、上传和展示结果。
- 云端完成深度估计、自动分割、3D scaffold、体积计算和热量估计。
- 单次图片推理在热启动状态下尽量控制在 10 秒左右。

## 2. 前端负责内容

小程序前端只保留必要输入：

- 拍摄或上传一张 RGB 食物图片。
- 可选输入食物类别，后续可由识别模型自动补齐。
- 可选读取手机型号，传给后端做相机参数匹配。
- 展示后端返回的体积、质量、热量和可视化图。

前端不再要求用户手动上传 mask，也不要求用户输入相机内参。

## 3. 后端负责内容

云端后端承担完整推理链路：

- 自动食物检测与分割，生成食物 mask。
- 读取手机型号、EXIF 或 DepthPro 焦距预测，推断相机内参。
- 调用 DepthPro 或后续替代 metric depth 模型，得到绝对尺度深度。
- 基于 mask、depth、intrinsics 生成 3D scaffold。
- 使用支撑平面方式估算体积。
- 结合密度与热量先验，输出质量和 kcal。
- 保存中间图像，便于调试和汇报。

## 4. 当前链路

```text
小程序图片
  -> 云端自动 mask
  -> 云端相机参数推断
  -> DepthPro metric depth
  -> 3D scaffold
  -> 支撑平面积分体积
  -> 食物密度 / 热量先验
  -> 体积、质量、kcal
```

## 5. 可优化点

优先级从高到低：

1. 自动分割稳定性：当前复杂中餐、汤面、盖浇饭、混合拼盘仍可能分割不准，需要引入更适合食物场景或开放词表的分割模型。
2. 深度模型速度：DepthPro 精度较好，但权重约 1.9GB，适合云端验证，不适合端侧部署。
3. 支撑平面鲁棒性：盘子、桌面区域识别越稳定，体积计算越可靠。
4. 食物类别先验：热量估计高度依赖食物类别、密度和 kcal/100g 先验，后续需要扩充中式菜肴先验库。
5. 服务化接口：需要把命令行脚本固定为云端 HTTP 接口，并记录耗时、错误、输出路径。

## 6. 本周建议验收指标

- 单张图片完整推理热启动耗时：目标小于 10 秒。
- 自动 mask 成功率：在 20 张常见食物图片上统计成功/失败。
- 体积输出稳定性：同一食物不同角度拍摄，体积波动尽量收敛。
- 后端接口稳定性：小程序只传图片也能得到结构化 JSON。
- 失败可解释性：分割失败、深度失败、支撑平面失败时能返回明确错误。
